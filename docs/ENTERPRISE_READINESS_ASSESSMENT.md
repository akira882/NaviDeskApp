# NaviDeskApp Enterprise納品可否評価レポート

**評価日**: 2026年3月15日
**対象**: NJC (従業員800人規模) への社内マニュアルHP納品
**評価者**: ドリーム・チーム全員 (Chief Test Engineer, Security Architect, Knowledge Operations Lead)

---

## エグゼクティブサマリー

### 最終結論

**現状のNaviDeskAppは、NJC (従業員800人規模) に『そのまま』納品することは不可能です。**

しかし、**明確な本番化ロードマップ (2.5-4週間) の実装を完了すれば、Enterprise級の社内マニュアルHPとして納品可能です。**

### 評価スコア

| 観点 | 現状評価 | 本番化後評価 | Gap |
|-----|---------|------------|-----|
| **セキュリティ** | ⚠️ 40/100 (MVP水準) | ✅ 90/100 (Enterprise水準) | 50pt |
| **品質管理** | ✅ 85/100 (テスト完備) | ✅ 95/100 (CI/CD完備) | 10pt |

**総合スコア**: 現状 45/100 → 本番化後 91/100

### Security Architect の視点

#### 評価: セキュリティ設計は優れているが、実装が未完成 ⚠️

**肯定的評価:**
1. **Server Components による境界明確化**
   ```typescript
   // lib/server/session.ts (1行目に "server-only")
   export function getSessionRole(): Role {
     const user = authProvider.getCurrentUser();
     return user.role;
   }
   ```
   - サーバー側でのみロール解決、クライアントバンドルへの混入を防止

2. **ロール別データフィルタリング**
   ```typescript
   // lib/server/initial-state.ts
   function filterStateForReader(role: Role): PortalContentState {
     return createInitialPortalState({
       articles: articles.filter(
         (article) => isApprovedForReaders(article) && canAccess(role, article.visibilityRole)
       ),
       // unauthorized contentはブラウザへ送らない
     });
   }
   ```
   - Phase 0改善により、クライアントコンポーネントがリポジトリを直接importしない設計

3. **セキュリティヘッダーの設定**
   - X-Frame-Options: DENY (クリックジャッキング対策)
   - X-Content-Type-Options: nosniff (MIME sniffing対策)
   - Cross-Origin-Opener-Policy: same-origin (Spectre対策)

**致命的な欠陥 (本番運用不可):**

#### 1. 認証基盤が存在しない ❌

```typescript
// lib/auth/mock-auth-provider.ts
export class MockAuthProvider implements AuthProvider {
  getCurrentUser(): User {
    const role = (env.NAVIDESK_SESSION_ROLE ?? "employee") as Role;
    // ↑ 環境変数で全ユーザーが同一ロール = 認証が存在しない
    return mockUsers.find((u) => u.role === role) ?? mockUsers[0];
  }
}
```

**問題点:**
- **全ユーザーが同一人物として扱われる** (環境変数 `NAVIDESK_SESSION_ROLE=admin` なら、全員がadmin)
- ログイン画面が存在しない
- セッション管理が存在しない
- パスワード認証が存在しない

**影響:**
- 800人の従業員を識別できない
- 監査ログの「誰が」が意味を成さない
- 管理職限定情報の制御が不可能

**解決策 (Phase 1):**
- SSO/OIDC統合 (NextAuth.js + Azure AD / Entra ID)
- 所要時間: 2-3日

#### 2. データ永続化が存在しない ❌

```typescript
// app/components/content-provider.tsx (クライアント側)
const [articles, setArticles] = useState<Article[]>(initialState.articles);
// ↑ ブラウザメモリのみ = リロードで消える
```

**問題点:**
- **リロードで全データが消える**
- **他ユーザーとデータ共有できない**
- **バックアップ不可能**

**影響:**
- 管理者が記事を更新しても、他の799人には反映されない
- ブラウザを閉じると、全更新が消える
- データベースバックアップ、リストア、監査が不可能

**解決策 (Phase 2):**
- PostgreSQL + Prisma ORM
- 所要時間: 3-5日

---

#### 3. 監査ログが改ざん可能 ❌

```typescript
// lib/content-helpers.ts (クライアント側で監査ログ生成)
export function updateArticle(
  state: PortalContentState,
  articleId: string,
  updates: Partial<Article>,
  actorId: string
): PortalContentState {
  // ...
  const auditLog = buildAuditLog({
    actorId, // ← クライアント側で actorId を生成 = 偽装可能
    action: "update",
    targetType: "article",
    targetId: articleId,
    detail: `更新: ${updates.title || article.title}`
  });
  // ↑ ブラウザ DevTools で改ざん可能
}
```

**問題点:**
- **監査ログをクライアント側で生成** = ブラウザDevToolsで改ざん可能
- **actorId を偽装可能** = 他人になりすまして操作記録を残せる
- **タイムスタンプを改ざん可能** = 操作時刻を偽装できる

**影響:**
- コンプライアンス監査で無効
- 内部不正の追跡不可能
- 法的証拠能力ゼロ

**解決策 (Phase 3):**
- サーバーサイド監査ログ (PostgreSQL + IP/User-Agent記録)
- 所要時間: 2-3日

---

#### 4. 静的ルート生成で非公開記事が露出 ⚠️

```typescript
// app/articles/[slug]/page.tsx
export async function generateStaticParams() {
  return articles.map((article) => ({ slug: article.slug }));
  // ↑ 全記事 (draft含む、manager-only含む) のslugを列挙
}
```

**問題点:**
- **非公開記事のslugがビルド時に露出**
- **管理職限定記事のslugが従業員に見える**

**影響 (現状は軽微だが改善必要):**
- ビルド成果物に全記事のslug一覧が含まれる
- 直接アクセスしてもコンテンツは表示されないが、slugの存在は推測可能

**解決策 (Phase 4):**
- 公開済み記事のみ `generateStaticParams()` で列挙
- または動的レンダリング (SSR) へ移行
- 所要時間: 1日
