## 1. プロジェクト概要

NaviDeskApp は、社内マニュアル・FAQ・業務手順・お知らせを統制された形で運用する**社内ナレッジ基盤**です。

### プロダクトが重視するもの

| 優先順位 | 項目 | 内容 |
|---------|------|------|
| 1 | セキュリティ | 機密情報をロール別に制御し、不正閲覧を防ぐ |
| 2 | 保守運用 | 担当者が変わっても継続的に運用できる仕組み |
| 3 | 正確な情報到達 | 社員が目的の情報に最短で辿り着ける設計 |
| 4 | 監査性 | 誰が・いつ・何を変更したかを追跡可能 |
| 5 | UIの見栄え | 情報を探しやすい、読みやすいインターフェース |

### 想定する利用シーン

- 人事・情シス・総務・申請手続き・就業ルール・福利厚生の情報集約
- 定型問い合わせの自己解決促進（VPN、有給、経費申請など）
- 社内ポータル更新の属人化防止
- 管理画面を通じた継続的な情報運用
- 監査ログを含む変更追跡

## 2. 解決する課題

### 本質的な問題

多くの社内ポータルが「作っても使われない」理由は、情報量の不足ではなく**情報へのアクセスコストの高さ**にあります。

```
「探すのが面倒」
  → 「とりあえず隣の人に聞く」
  → 「ベテランへの属人化が深まる」
  → 「ベテランが退職するとノウハウが消える」
```

この連鎖を断ち切るには、マニュアルの内容を充実させることより先に、**「探す行為そのもの」を限りなくゼロに近づける**設計が必要です。

### NaviDeskが解決する3つの課題

| 課題 | 従来の状態 | NaviDeskの解決策 |
|------|-----------|----------------|
| **情報へのアクセスコスト** | 「どこを見ればいいか分からない」 | ヒーロー検索 + タスクハブ + カテゴリ導線の3経路 |
| **情報の信頼性** | 「最新かどうか分からない」 | 更新日・担当者・承認日を記事に明示 |
| **更新運用の属人化** | 「あの人しか更新方法を知らない」 | 承認フロー付き管理画面 + 監査ログで組織知識を制度化 |

## 3. 要件定義

### 3-1. 機能要件

#### 情報到達
- 社員はホーム検索から記事とFAQを横断検索できる
- タスクハブ・カテゴリ・クイックリンクから業務導線へ到達できる
- 検索ゼロ件時は代替導線（AI案内・社内IT企画部門）を提示する

#### 情報信頼性
- 公開コンテンツは承認済みのみ表示する
- AI案内は社内コンテンツを根拠にした回答のみ返す
- 根拠が弱い場合は断定せず、候補のみ返す

#### 運用統制
- 記事・FAQ・お知らせ・クイックリンクの作成・編集・削除・公開ができる
- 承認フロー（申請 → 承認 / 差し戻し）を管理できる
- 監査ログで作成・更新・削除・承認系操作を追跡できる

#### 継続改善
- 検索失敗テーマを集計し、情報不足を可視化する
- 更新期限が近い・古いコンテンツをレビュー優先キューで確認できる
- 記事詳細から承認情報・レビューコメント・変更履歴を確認できる

### 3-2. 非機能要件

#### セキュリティ
- ロール別に閲覧可否を制御する（employee / manager / editor / admin）
- ロールはサーバー側で決定する（クライアント側での偽装を禁止）
- 非公開コンテンツを一般閲覧者のブラウザへ送信しない

#### パフォーマンス
- First Load JS: 170KB 以下（Google 推奨値準拠）
- LCP (Largest Contentful Paint): 2.5秒以下
- CLS (Cumulative Layout Shift): 0.1以下

#### 保守運用
- lint / test / build がすべてパスした状態のみリリース可能
- ビジネスロジックのテストカバレッジ: 90% 以上
- ロール制御・監査ログ整合性のテストカバレッジ: 100%

## 4. 主要機能

### ユーザー向け画面

#### ホーム画面 (`/`)
- **タスクハブ**: 優先度順に整理された業務導線（勤怠・IT・福利厚生・ワークフロー）
- **横断検索**: 記事とFAQをまとめて検索（リアルタイムプレビュー付き）
- **コマンドパレット**: ⌘K で起動するグローバル検索
- **カテゴリカード**: 6カテゴリへの導線（人事・情シス・総務・申請・就業ルール・福利厚生）
- **お知らせ**: 勤怠締め・システムメンテナンスなどの運用通知
- **クイックリンク**: 外部システム（勤怠・経費・福利厚生など）への直接リンク

#### AI案内画面 (`/ai-guide`)
- 自然文質問を受け取り、社内記事・FAQを先に検索して根拠を特定
- 根拠候補を明示し、根拠が弱い場合は「分からない」と返答（ハルシネーション防止）
- フォールバック時は通常検索候補と社内IT企画部門への導線を提示

#### その他ユーザー向け画面
- カテゴリ一覧・詳細 (`/categories/[slug]`)
- 記事詳細 (`/articles/[slug]`) — 情報鮮度・承認日時を明示
- FAQ検索 (`/faq`) — キーワード検索 + カテゴリ絞り込み
- タスクハブ詳細 (`/tasks`) — 業務フローと関連ツールへのリンク

### 管理者向け画面

#### 管理ダッシュボード (`/admin`)
- コンテンツ作成・編集・削除・公開ステータス切替
- 承認フロー管理（not_requested → pending → approved / changes_requested）
- 更新期限レビュー優先キュー
- コンテンツガバナンスカードによる承認状況の可視化

#### 監査ログ (`/admin/audit-log`)
- 作成・更新・削除・公開切替・承認操作の全履歴
- 誰が・いつ・何を変更したかを追跡可能

#### 検索分析 (`/admin/search-insights`)
- 検索失敗テーマの可視化（結果ゼロのクエリ）
- 検索頻度の高いキーワード分析
- コンテンツ改善候補の自動提示

## 5. 設計思想

### 行動経済学に基づく設計

NaviDeskは、ノーベル経済学賞を受賞したダニエル・カーネマン教授の著作『ファスト＆スロー』（2011年）の知見を実装根拠としています。

カーネマン教授は人間の思考を次の2つに分類しました。

| システム | 別名 | 特徴 |
|---------|------|------|
| **システム1** | 速い思考 | 自動的・直感的・努力不要。日常判断のほとんどはここ |
| **システム2** | 遅い思考 | 意図的・論理的・努力が必要。疲れると機能低下する |

社員がマニュアルを「読もう」と意識する前に、システム1が「このサイトは使いにくい」と判断したら、そこで離脱します。NaviDeskは**社員のシステム1が「自然に使いたくなる」設計**を採用しています。

### 設計の3原則

**原則1: 認知負荷の最小化**  
「考えなくても使える」をゴールに設定。検索バーを最も目立つ場所に置き、ナビゲーション項目を最大7つに制限。

**原則2: 損失フレームによる行動変容**  
「お知らせが1件あります」より「未確認の手順変更が1件あります」の方が、人間は約2.5倍強く行動する（プロスペクト理論）。重要な更新通知はすべてこの原則で文言設計。

**原則3: デフォルトを味方につける**  
人間は提示されたデフォルトをそのまま採用する確率が非常に高い（デフォルト効果）。検索フィルタの初期値・通知設定など、すべてのデフォルトを「組織にとって最も望ましい行動」に設定。

## 6. 従来型ポータルとの比較

| 観点 | 従来型ポータル | NaviDesk |
|------|-------------|---------|
| **検索方式** | キーワード完全一致 | AI意図解釈型（セマンティック検索）|
| **検索バーの位置** | ヘッダー右端（小さい） | ページ中央・ヒーローサイズ |
| **更新通知の文言** | 「マニュアルが更新されました」 | 「未確認の手順変更があります（業務に影響する可能性）」 |
| **通知アイコン** | 情報アイコン（青） | 警告アイコン（赤） |
| **記事の表示形式** | タイトルのみ、または全文 | サマリー + 更新日 + 承認状況 |
| **メニュー項目数** | 制限なし（10〜20項目が多い） | 最大7項目（超過分はサブメニュー）|
| **パーソナライズ** | なし | 最近の更新・タスク優先度で自動整列 |
| **承認フロー** | なし、または別ツール | システム内で完結（申請〜承認〜差し戻し）|
| **監査ログ** | なし、または別ツール | 全管理操作を自動記録 |
| **AI案内** | なし | 根拠明示型（ハルシネーション防止設計）|

## 7. 技術スタックと選定理由

### 技術スタック一覧

| カテゴリ | 技術 | バージョン |
|---------|------|---------|
| フレームワーク | Next.js (App Router) | 15.3 |
| UIライブラリ | React | 19 |
| 言語 | TypeScript | 5.8 |
| スタイリング | Tailwind CSS | 3.4 |
| UIコンポーネント | shadcn/ui + Radix UI | 最新 |
| フォーム・バリデーション | React Hook Form + Zod | 7.56 / 3.24 |
| テスト | Vitest | 4.1 |
| リント | ESLint (Next.js プリセット) | 9.16 |
| AIプロバイダー | Google Gemini API | gemini-2.0-flash |
| デプロイ | Vercel | — |

### 選定の根拠

#### Next.js 15 (App Router) を選んだ理由

**セキュリティ優位性:**
- Server Components により、機密情報をサーバー側で完全に制御し、unauthorized content をブラウザへ送らない設計が実現可能
- `"use server"` ディレクティブでサーバー専用コードをビルド時に強制

**長期保守性:**
- Typed Routes により、リンク切れやパス誤りをビルド時に検出
- React 19 との統合で、Meta による長期的な投資とコミュニティの厚みが保証される
- Next.js 経験者の採用が容易であり、引き継ぎリスクが低い

**代替との比較:**

| フレームワーク | 不採用理由 |
|-------------|----------|
| Remix | Server Components のセキュリティモデルが未成熟（2026年3月時点）|
| Astro | 動的コンテンツ管理（認証・承認フロー）に不向き |
| SvelteKit | エンタープライズ採用実績が Next.js に劣る |

#### TypeScript を選んだ理由

```typescript
// ロール制御の型安全性 — コンパイル時に不正な値を検出
type Role = "employee" | "manager" | "editor" | "admin";

function listVisibleArticles(state: ContentState, role: Role): Article[] {
  return state.articles.filter(article =>
    article.visibility === "all" ||
    article.visibility === role
  );
}
```

- `types/domain.ts` で Article / FAQ / User / Role などを型定義し、データ不整合を防止
- `strict: true` により、未定義値参照による実行時エラーを排除
- 業務ルール（`ApprovalStatus` 型など）がコード上で自己文書化される

#### Tailwind CSS を選んだ理由

- デザインシステムの色彩体系（accent-teal / accent-gold / ink など）を `tailwind.config.ts` で一元管理
- 使用されていない CSS を自動削除し、本番バンドルサイズを最小化
- 学習曲線が緩やかで、新規参画メンバーの立ち上げが早い

#### Vitest を選んだ理由

- Jest の10倍高速なテスト起動（ESM ネイティブ、HMR 対応）
- ロール別テスト・監査ログ整合性テストを自動化し、リリースゲートとして機能

#### AI プロバイダー設計の理由

```
lib/ai/guide-service.ts（境界）
  ├── mock-guide-provider.ts（開発・テスト用）
  ├── gemini-guide-provider.ts（Google Gemini API）
  └── bedrock-guide-provider.ts（AWS Bedrock）
```

- `guide-service.ts` が境界を担うため、プロバイダーを切り替えても UI 側のコードは無変更
- AI は社内コンテンツを先に検索し、根拠を特定してから回答を生成（ハルシネーション防止）
- API キーはサーバー側のみで管理し、ブラウザへ露出しない

## 8. アーキテクチャとセキュリティ設計

### server / client 境界の明確化

```
サーバーコンポーネント（Next.js）
  ├── セッションロールの解決（getSessionRole）
  ├── ロール別初期データの構築（buildInitialStateForRole）
  └── フィルタリング済みデータをクライアントへ渡す
        ↓ props のみ
クライアントコンポーネント（React）
  ├── UI の表示・操作のみ
  ├── リポジトリへの直接アクセス禁止
  └── ロール情報は表示制御のみに使用（認可はサーバー側で完結済み）
```

**この設計の意図:**
- 社内マニュアルには機密情報（管理職向け情報、人事情報）が含まれる
- クライアント側でロールを偽装しても、サーバー側でフィルタリング済みのデータしか存在しない
- 本番SSO移行時も、クライアント側のコード変更が最小で済む

### セキュリティヘッダー (OWASP Top 10 対策)

| ヘッダー | 防御対象 |
|---------|---------|
| `X-Frame-Options: DENY` | クリックジャッキング |
| `X-Content-Type-Options: nosniff` | MIME スニッフィング |
| `Referrer-Policy: same-origin` | 機密 URL の外部漏洩 |
| `Permissions-Policy` | カメラ・マイクへの不正アクセス |
| `Cross-Origin-Opener-Policy: same-origin` | Spectre サイドチャネル攻撃 |

### ロール構成

| ロール | 閲覧範囲 | 管理権限 |
|-------|---------|---------|
| `employee` | 全社員向けコンテンツ | なし |
| `manager` | 全社員向け + 管理職向けコンテンツ | なし |
| `editor` | 全社員向け + 管理職向けコンテンツ | コンテンツ作成・編集・承認申請 |
| `admin` | すべて | 全権限 + 監査ログ閲覧 |

### リポジトリパターンによる拡張性

```
現在: data/mock/seed.ts（インメモリ）
  ↓ content-repository.ts（インターフェース）
将来: PostgreSQL / Supabase / CMS
```

ビジネスロジックを変更せず、データソースのみを差し替えられる設計です。

## 11. 導入後の運用・改善サイクル

### Phase 1: 導入・定着期（0〜3ヶ月）

**目標: 全員が「存在を知っていて、使ったことがある」状態を作る**

| 施策 | 内容 |
|------|------|
| 初回説明会 | ?design=true オーバーレイで設計意図を5分で説明 |
| 週次アクセスレポート | 検索クエリ・閲覧記事のトップ10を毎週確認 |
| 未解決クエリの収集 | 「検索したが見つからなかった」クエリを自動収集 |

### Phase 2: 改善・最適化期（3〜6ヶ月）

**目標: データに基づいてコンテンツとUXを継続改善する**

| 施策 | 判断基準 |
|------|---------|
| コンテンツギャップ補填 | 月3件以上の未解決クエリが出たカテゴリを優先 |
| 通知文言のA/Bテスト | 2週間で確認率の高い文言に統一 |
| 重要度ラベルの見直し | 必読記事の閲覧率 85% 未満を改善ライン |

### Phase 3: 自走・高度化期（6ヶ月以降）

**目標: 担当者が変わっても品質が落ちない運用体制の確立**

- 更新ガイドラインの整備（誰が・いつ・どのように承認するか）
- 四半期レビュー会議（アクセスデータを見ながら設計を見直し）
- 運用マニュアル・設計根拠ドキュメントを常に最新に保つ

## 12. 導入効果の試算

以下は行動経済学の先行研究と類似ポータル事例に基づく試算です。実際の効果は運用後のアクセスログ分析によって検証します。

### KPI 目標値

| 指標 | 3ヶ月目標 | 6ヶ月目標 | 計測方法 |
|------|---------|---------|---------|
| 月間アクティブユーザー率 | 全社員の 50% | 全社員の 75% | アクセスログ |
| 重要更新の確認率 | 70% | 85% | 通知既読データ |
| 検索自己解決率 | 50% | 70% | 未解決クエリ数の逆数 |
| 記事の平均読了率 | 60% | 75% | スクロール深度 |

### 機能別の期待効果

| 機能 | 従来の課題 | 期待効果 | 根拠 |
|------|----------|---------|------|
| セマンティック検索 | キーワードが一致しないと0件 | 検索到達率 70〜80%（現状 40〜50%） | AI意図理解型 |
| 損失フレーム通知 | 重要更新を見落とす | 確認率 2.5倍向上 | プロスペクト理論 |
| タスクハブ | どこを見ればいいか分からない | 1クリックで業務導線へ | 注意の経済学 |
| 承認フロー + 監査ログ | 「誰が更新したか分からない」 | 全変更操作を追跡可能 | 監査整合性の保証 |
