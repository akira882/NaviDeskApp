# NaviDeskApp

NaviDeskApp は、社内マニュアル、FAQ、業務手順、お知らせを統制された形で運用するための社内ナレッジ基盤です。  
主目的は、社員が正しい情報へ短時間で到達できることと、運用担当者が安全に保守し続けられることです。

## このプロダクトが重視するもの

- 情報を探しやすいこと
- 更新責任が曖昧にならないこと
- 誰に見せる情報かを制御できること
- 変更履歴を追えること
- 将来の認証基盤、DB、AI連携へ安全に拡張できること

## 想定する利用シーン

- 人事、情シス、総務、申請手続き、就業ルール、福利厚生の案内集約
- 定型問い合わせの自己解決促進
- 社内ポータル更新の属人化防止
- 管理画面を通じた継続運用
- 監査ログを含む更新追跡

## 主要画面と機能

### ユーザー向け画面

#### ホーム画面 (`/`)
- **タスクハブ**: 優先度順に整理された業務導線（勤怠、IT、福利厚生、ワークフロー）
- **横断検索**: 記事とFAQをまとめて検索し、自己解決を促進
- **カテゴリカード**: 6カテゴリ（人事、情シス、総務、申請・手続き、就業ルール、福利厚生）への導線
- **よく見られている記事**: 最近更新された重要情報を優先表示
- **お知らせ**: 勤怠締め、システムメンテナンスなどの運用通知
- **クイックリンク**: 外部システム（勤怠、経費、福利厚生など）への直接リンク
- **ITサポートデスクへの問い合わせ導線**: 検索失敗時のエスカレーション

#### カテゴリ一覧・詳細画面 (`/categories`, `/categories/[slug]`)
- カテゴリ別に記事を一覧表示
- 各記事の要約、タグ、更新日を表示し、情報の鮮度を可視化

#### 記事詳細画面 (`/articles/[slug]`)
- タイトル、要約、本文、タグ、関連記事を表示
- **情報鮮度セクション**: 最終更新日、更新担当者、レビュー承認日時を明示
- **問い合わせ導線**: 記事で解決しない場合のITサポートデスクへの誘導
- ロール別表示制御（employee、manager、editor、admin）

#### FAQ検索画面 (`/faq`)
- キーワード検索とカテゴリ絞り込み
- 検索失敗時の代替導線（カテゴリ、AI案内、ITサポートデスク）

#### AI案内画面 (`/ai-guide`)
- 自然文質問を受け取り、社内記事・FAQを先に検索
- 根拠候補を明示し、根拠が弱い場合は「分からない」と返答
- フォールバック時は通常検索候補を表示
- 候補ゼロ時はITサポートデスクへの導線を提示

#### タスクハブ画面 (`/tasks`)
- 優先度順に業務フローを整理（勤怠、IT、福利厚生、ワークフロー）
- 各タスクの詳細説明と関連ツールへのリンク

### 管理者向け画面

#### 管理画面 (`/admin`)
- 記事、FAQ、お知らせ、クイックリンクの作成・編集・削除
- 公開ステータス切替（draft ⇄ published）
- 承認フロー管理（not_requested → pending → approved/changes_requested）
- 更新期限レビュー優先キュー
- コンテンツガバナンスカードによる承認状況の可視化

#### 監査ログ画面 (`/admin/audit-log`)
- 作成、更新、削除、公開切替、承認操作の履歴確認
- 誰が、いつ、何を変更したかを追跡可能

#### 検索分析画面 (`/admin/search-insights`)
- 検索失敗テーマの可視化（結果ゼロのクエリ）
- 検索頻度の高いキーワード分析
- コンテンツ改善候補の提示

## ITサポートデスク業務への適合性

NaviDeskAppは、ITサポートデスクの業務フローに最適化された設計になっています：

### 段階的エスカレーション設計
1. **自己解決の促進**: ホーム検索 → FAQ → AI案内 → 記事詳細
2. **エスカレーション導線**: 解決しない場合のITサポートデスクへの明示的な誘導
3. **問い合わせ前の情報収集**: ヘルプデスク記事で症状、発生時刻、影響範囲の確認を促す

### 定型問い合わせのカバレッジ
- VPN接続、パスワードリセット、勤怠、有休、経費など、実際のヘルプデスクで多い問い合わせを18件の記事でカバー
- 入社から退職までのライフサイクル対応
- 申請フロー（休暇、経費、出張、残業、備品）の網羅

### 情報鮮度の可視化
- 各記事に最終更新日・更新担当者を明示
- 運用担当者が継続的に情報を更新できる仕組み
- 古い情報による問い合わせ増加を防止

### 運用負荷の最小化
- 問い合わせ導線を統一（すべてヘルプデスク記事へ集約）
- 検索分析による改善候補の可視化
- 監査ログによる変更追跡と責任の明確化

## ロール

- `employee`: 一般社員向け情報を閲覧
- `manager`: 管理職向け情報を含めて閲覧
- `editor`: コンテンツ管理
- `admin`: 全権限と監査ログ閲覧

セッションロールはサーバー側で解決します。クライアント側でのロール切替は許可しません。

## Dream Team

- `Chief Test Engineer`: lint、test、build をリリースゲートとして管理し、リファクタリング時の回帰を先に検出する
- `Security Architect`: client/server 境界と機密情報露出の後退を防ぐ
- `Knowledge Operations Lead`: 情報設計、検索性、更新責任の明確さを維持する

## 保守運用の考え方

### 1. 社内コンテンツを唯一の情報源として扱う

AI は制度や手順を新規に作らず、社内記事と FAQ を先に検索して案内します。根拠が弱い場合は断定せず、候補のみ返します。

### 2. 管理操作と監査を分離しない

作成、編集、削除、公開切替は、ポータル全体への反映と監査ログ更新が同時に成立して初めて正しい運用とみなします。

### 3. 保守しやすい境界を守る

- 画面: `app/`
- UI コンポーネント: `app/components/`
- データ参照: `data/`
- ビジネスロジック: `lib/`
- ドメイン型: `types/`
- 運用文書: `docs/`

UI から直接ベタ書きデータを触らず、型と helper を通して整合性を保ちます。

### 4. セキュリティを保守要件として扱う

このアプリは社内マニュアルを扱うため、セキュリティは後付けではなく保守運用そのものです。

- 機密情報は server-side only
- unauthorized content は初期描画時点でブラウザに送らない
- session role はサーバー側で決定
- AI provider key はクライアントへ露出しない

詳細は [docs/SECURITY.md](./docs/SECURITY.md) を参照してください。

## 設計判断と理由

### なぜこのアーキテクチャにしたか

#### 1. client/server 境界の明確化

**判断**: サーバーサイドで認証・認可を解決し、unauthorized content をブラウザへ送らない

**理由**:
- 社内マニュアルには機密情報（管理職向け情報、人事情報など）が含まれる
- クライアント側ロール切替は開発時の利便性は高いが、セキュリティリスクが大きい
- 本番SSO移行時にクライアント側のコード変更を最小化できる

**実装**:
- `lib/server/session.ts`でサーバー側セッションロール解決
- `lib/server/initial-state.ts`でロール別に絞った初期データをクライアントへ渡す
- `"use server"`ディレクティブで server-only コードを明示

#### 2. AI は社内コンテンツの要約補助であり、制度の決定者ではない

**判断**: AI は記事・FAQを先に検索し、根拠が弱い場合は「分からない」と返す

**理由**:
- 社内制度や手順をAIが創作すると、誤った情報提供のリスクがある
- ITサポートデスクの責任は「正確な情報への誘導」であり、「AIによる回答生成」ではない
- 根拠候補を明示することで、ユーザーが自分で判断できる

**実装**:
- `lib/ai/guide-service.ts`で provider境界を定義
- 現在は`mock`、将来は`gemini`など切替可能
- 回答と根拠候補を分離し、根拠不足時はフォールバックモードへ

#### 3. 管理操作と監査を分離しない

**判断**: 作成、編集、削除、公開切替は、監査ログ更新と同時に成立して初めて正しい運用とみなす

**理由**:
- 社内マニュアルの変更は「誰が、いつ、何を変更したか」の追跡が必須
- 監査ログが後付けだと、ログ更新漏れのリスクがある
- コンプライアンスや運用監査の観点で、変更履歴が重要

**実装**:
- `lib/content-helpers.ts`の管理操作関数内で監査ログ生成を一体化
- `lib/audit.ts`で監査ログ生成ロジックを集約
- ContentProviderで共有状態として管理し、すべての操作を追跡

#### 4. 保守しやすい境界を守る

**判断**: UI から直接ベタ書きデータを触らず、型と helper を通して整合性を保つ

**理由**:
- 直接データ操作はバグの温床になる
- ロジック変更時の影響範囲が広がる
- 引き継ぎ時に「どこで何をしているか」が分からなくなる

**実装**:
- `types/domain.ts`で型定義を一元管理
- `lib/content-helpers.ts`でビジネスロジックを集約
- UIコンポーネントはロジックを含まず、データ表示に専念

#### 5. 段階的エスカレーション設計

**判断**: 自己解決を促進し、解決しない場合のみITサポートデスクへエスカレーション

**理由**:
- ITサポートデスクの業務負荷を下げるには、定型問い合わせの自己解決が重要
- ユーザーが迷子にならないよう、次のアクションを明示する必要がある
- 問い合わせ導線を統一することで、運用負荷が最小化される

**実装**:
- ホーム、FAQ、AI案内、記事詳細の4箇所にITサポートデスク導線を配置
- すべてヘルプデスク記事（`/articles/helpdesk-contact`）へ集約
- 問い合わせ前に「症状、発生時刻、影響範囲」の確認を促す

### 将来拡張を考慮した設計

#### 認証基盤の切替可能性
- `lib/auth/auth-provider.ts`でインターフェース定義
- 現在は`MockAuthProvider`、将来は`SSOAuthProvider`へ切替
- `lib/server/session.ts`の変更のみで対応可能

#### AI provider の切替可能性
- `lib/ai/guide-service.ts`で provider境界を定義
- 現在は`mock`、将来は`gemini`へ切替
- クライアント側コードの変更不要

#### データソースの切替可能性
- `data/repositories/content-repository.ts`でデータアクセス層を抽象化
- 現在は`seed.ts`、将来はDB・CMSへ切替
- ビジネスロジックの変更不要

#### 承認フローの拡張性
- `types/domain.ts`で`ApprovalStatus`型定義
- ContentGovernanceCardで承認状況の可視化
- 将来の多段階承認にも対応可能

## 現在の段階と限界

⚠️ **このプロジェクトは現在MVP/プロトタイプ段階です**

以下の機能は**意図的に簡略化**されており、本番運用には**追加の実装が必要**です:

### 認証・認可
- **現状**: 環境変数による固定ロール（全ユーザーが同一ロール）

- **現状**: ブラウザメモリのみ（リロードで消える）

### 監査ログ
- **現状**: クライアント側生成（改ざん可能）
- **本番要件**: サーバー側記録、改ざん耐性

### 本番化への移行計画

詳細な実装ロードマップは **[docs/PRODUCTION_ROADMAP.md](./docs/PRODUCTION_ROADMAP.md)** を参照してください。

- Phase 4: セキュリティ強化（1-2日）

## 引き継ぎ時に重要なこと

- UI を直すだけでは不十分で、ロール制御、監査ログ、AI フォールバックまで確認する
- セキュリティ境界を client 側へ戻さない
- 更新運用を壊す変更は docs も同時に更新する
- 「便利そう」より「保守しやすい」を優先する
