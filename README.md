# NaviDeskApp

NaviDeskApp は、社内マニュアル、FAQ、業務手順、お知らせを統制された形で運用するための社内ナレッジ基盤です。  
主目的は、社員が正しい情報へ短時間で到達できることと、運用担当者が安全に保守し続けられることです。

## このプロダクトが重視するもの

- 情報を探しやすいこと
- 更新責任が曖昧にならないこと
- 誰に見せる情報かを制御できること
- 変更履歴を追えること
- 将来の認証基盤、DB、AI連携へ安全に拡張できること

## 想定する利用シーン

- 人事、情シス、総務、申請手続き、就業ルール、福利厚生の案内集約
- 定型問い合わせの自己解決促進
- 社内ポータル更新の属人化防止
- 管理画面を通じた継続運用
- 監査ログを含む更新追跡

## 現在の機能

- ホームからの記事・FAQ横断検索
- カテゴリ別の業務情報閲覧
- 記事詳細と関連情報表示
- FAQ検索とカテゴリ絞り込み
- 社内コンテンツ根拠ベースの AI 案内
- 記事、FAQ、お知らせ、クイックリンクの管理
- 監査ログの確認
- ロール別表示制御

## ロール

- `employee`: 一般社員向け情報を閲覧
- `manager`: 管理職向け情報を含めて閲覧
- `editor`: コンテンツ管理
- `admin`: 全権限と監査ログ閲覧

セッションロールはサーバー側で解決します。クライアント側でのロール切替は許可しません。

## 保守運用の考え方

### 1. 社内コンテンツを唯一の情報源として扱う

AI は制度や手順を新規に作らず、社内記事と FAQ を先に検索して案内します。根拠が弱い場合は断定せず、候補のみ返します。

### 2. 管理操作と監査を分離しない

作成、編集、削除、公開切替は、ポータル全体への反映と監査ログ更新が同時に成立して初めて正しい運用とみなします。

### 3. 保守しやすい境界を守る

- 画面: `app/`
- UI コンポーネント: `app/components/`
- データ参照: `data/`
- ビジネスロジック: `lib/`
- ドメイン型: `types/`
- 運用文書: `docs/`

UI から直接ベタ書きデータを触らず、型と helper を通して整合性を保ちます。

### 4. セキュリティを保守要件として扱う

このアプリは社内マニュアルを扱うため、セキュリティは後付けではなく保守運用そのものです。

- 機密情報は server-side only
- unauthorized content は初期描画時点でブラウザに送らない
- session role はサーバー側で決定
- AI provider key はクライアントへ露出しない

詳細は [docs/SECURITY.md](./docs/SECURITY.md) を参照してください。

## アーキテクチャ概要

### 主要ファイル

- `types/domain.ts`: ドメインモデル
- `data/mock/seed.ts`: サンプルデータ
- `data/repositories/content-repository.ts`: 基礎データ参照
- `lib/content-helpers.ts`: 可視性、検索、監査補助
- `lib/ai/guide-service.ts`: AI provider 境界
- `lib/server/session.ts`: サーバー側セッションロール
- `lib/server/initial-state.ts`: ロール別初期データ構築
- `app/components/content-provider.tsx`: 管理操作の共有状態

### 現在のランタイム

現在は保守運用の流れを確認しやすくするため、管理対象コンテンツを共有ストアで保持しています。  
ただし初期状態はサーバー側でロール別に絞ってからクライアントへ渡しており、危険なクライアント側ロール切替や browser storage 永続化は行っていません。

- UI を直すだけでは不十分で、ロール制御、監査ログ、AI フォールバックまで確認する

- 更新運用を壊す変更は docs も同時に更新する
- 「便利そう」より「保守しやすい」を優先する
