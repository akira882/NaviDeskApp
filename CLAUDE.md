# NaviDeskApp

## 目的

- このアプリは社内マニュアルと業務ナレッジの運用基盤である。
- 最優先は新機能追加ではなく、保守しやすさ、誤更新防止、監査性、検索性である。
- 変更は必ず「誰が、どこを、なぜ直すのか」が追える状態を保つ。

## 基本方針

- 既存の責務分離を崩さない。
- 画面だけ直してデータ境界や監査整合性を壊さない。
- 推測で仕様を広げない。
- 大きな抽象化より、引き継ぎしやすい明示的なコードを優先する。

## 優先順位

1. セキュリティ
2. 保守運用
3. 正確な情報到達
4. 監査性
5. UIの見栄え

## 読む順序

- 仕様と前提: `README.md`
- セキュリティ境界: `docs/SECURITY.md`
- 運用観点: `docs/OPERATIONS.md`
- サーバー側境界: `lib/server/*`
- ドメイン型: `types/domain.ts`
- データと参照元: `data/mock/seed.ts`, `data/repositories/content-repository.ts`
- ビジネスロジック: `lib/content-helpers.ts`, `lib/ai/*`
- UIと画面: `app/*`, `app/components/*`

## 変更時のルール

- 記事、FAQ、お知らせ、クイックリンクを触る変更では、監査ログへの影響を確認する。
- ロール制御を触る変更では、`employee`, `manager`, `editor`, `admin` の見え方を確認する。
- AI案内を触る変更では、根拠不足時のフォールバックを壊さない。
- 環境変数を増やす場合は `.env.example` と `lib/env.ts` を同時更新する。
- サーバー側で扱うべき情報を client component へ移さない。

## 禁止事項

- クライアント側ロール切替の再導入
- 機密情報の browser storage 保存
- API key や秘密情報の client 露出
- 根拠なしの AI 回答生成
- repository や helper を通さないベタ書きデータ参照
- 監査ログを更新しない管理操作の追加

## 実装基準

- 型安全を崩さない。
- 小さな pure function を優先する。
- JSX 内に複雑な業務判断を埋め込まない。
- 命名は業務上の意味が分かるものにする。
- コメントは必要最小限にし、運用判断が読み取りにくい箇所だけ補う。

## 保守運用基準

- 修正は最小差分で行う。
- 影響範囲が広い変更は README または docs を更新する。
- 仕様変更を伴う場合は、画面、データ、運用の3点が揃っているか確認する。
- 不具合修正では、再発防止のためテスト追加を優先する。

## AI案内の扱い

- 現在の provider は `mock` である。
- 将来の `gemini` 追加時も `lib/ai/guide-service.ts` を境界に保つ。
- AI は社内コンテンツの要約補助であり、制度の決定者ではない。
- 根拠候補、回答、フォールバックの3つを常に分けて考える。

## 開発ツール (Claude Code CLI利用時)

- 使用するモデルは **Sonnet** を基本とし、実行時に適切なバージョンを選択すること。
- 修正や機能追加の際は、本ファイル (`CLAUDE.md`) のルールを常に遵守させる。

## 検証

- 通常変更: `npm run lint`
- ロジック変更: `npm run test`
- 画面、ルーティング、型変更: `npm run build`
- 大きな変更の完了前: `npm run lint && npm run test && npm run build`

## 完了条件

- 変更意図がコード上で説明可能
- セキュリティ境界が後退していない
- 監査性が壊れていない
- README か docs に必要な更新が反映されている
